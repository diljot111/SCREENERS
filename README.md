# Real-Time Indian Stock Screener with WhatsApp Alerts

Monitors **all NSE listed stocks** in real time during market hours, applies a
technical-indicator crossover filter on the **daily (1D) timeframe**, and sends
**WhatsApp** alerts (via [Baileys](https://github.com/WhiskeySockets/Baileys))
when a stock matches.

## The signal

Both conditions must be true on the daily timeframe for an alert to fire:

1. **9 EMA crossed above** the Bollinger Band **middle** line, and
2. **VWAP crossed above** the Bollinger Band **upper** line.

A "crossover" means: yesterday's value was `<=` the band and the current
(forming-candle) value is `>` the band. Yesterday's indicator values are
pre-computed once at market open and cached; today's forming values are
recomputed every 2 minutes from the closed-candle history plus the live candle
built from WebSocket ticks.

## Architecture

```
stock-screener/
├── config/
│   ├── config.json          # API keys, phone number, limits (edit this)
│   └── nse_symbols.json      # NSE symbol master (generated by a script)
├── python-engine/           # The screener (Python)
│   ├── main.py               # Orchestrator / entry point (APScheduler)
│   ├── data_fetcher.py       # Angel One auth, historical, live WebSocket
│   ├── candle_builder.py     # Builds today's forming candle from ticks
│   ├── indicator_engine.py   # BB, 9 EMA, VWAP
│   ├── screener.py           # Crossover filter
│   ├── alert_manager.py      # Dedup, 400/day cap, queue, retries
│   └── db.py                 # SQLite + config/logging helpers
├── whatsapp-service/        # Baileys WhatsApp bot (Node.js)
│   ├── index.js              # HTTP API + QR login + send queue
│   └── auth_store/           # Session (auto-created on first QR scan)
├── scripts/
│   ├── fetch_nse_symbols.py  # Download + filter NSE equity symbols
│   └── backfill_history.py   # Populate last 30 days of candles
├── database/screener.db     # SQLite (auto-created)
└── logs/screener.log        # Rotating log (auto-created)
```

Two processes run side by side:
- **Node WhatsApp service** (`whatsapp-service`) on `http://localhost:3001`.
- **Python screener** (`python-engine`) which POSTs alerts to that service.

## Prerequisites

- **Python 3.10+**
- **Node.js 18+**
- An **Angel One** trading account with SmartAPI enabled (free).
- A spare phone with WhatsApp for linking the bot.

---

## 1. Get Angel One API credentials

1. Sign up / log in at <https://smartapi.angelbroking.com/>.
2. Create an app under **My Profile → API** to get your **API key**.
3. Your **Client ID** is your Angel One login ID.
4. Your **Password** here is your **MPIN/PIN** used for SmartAPI login.
5. Enable **TOTP** and save the **TOTP secret** (the base32 string shown when
   you set up the authenticator). The screener generates the 6-digit code from
   this secret automatically with `pyotp` — no manual entry.

Put these in `config/config.json`:

```json
"angel_one": {
    "api_key": "...",
    "client_id": "...",
    "password": "...",
    "totp_secret": "..."
}
```

Also set your WhatsApp recipient number (country code + number, **no `+`**):

```json
"whatsapp": { "target_phone": "919876543210", ... }
```

> Tip: copy `config/config.json` to `config/config.local.json` and put your real
> secrets there. `config.local.json` overrides `config.json` and is git-ignored.

## 2. Install dependencies

```bash
# Python
cd python-engine
python -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
cd ..

# Node
cd whatsapp-service
npm install
cd ..
```

> **Note on indicators:** `pandas_ta` is listed in `requirements.txt` for
> convenience, but the engine computes Bollinger Bands / EMA directly with
> `pandas` (mathematically identical, and immune to `pandas_ta` breaking against
> newer `numpy`). If `pandas_ta` fails to install on your platform you can
> remove it from `requirements.txt` — the screener does not import it.

## 3. Download the NSE symbol list

```bash
python scripts/fetch_nse_symbols.py
```

This downloads Angel One's instrument master, keeps only NSE common-equity
(`-EQ`) symbols, and writes `config/nse_symbols.json`. (A small sample of 5
symbols ships in the repo so you can test before running this.)

## 4. Backfill historical candles (one-time / pre-market)

```bash
python scripts/backfill_history.py        # uses config history_lookback_days (30)
# or: python scripts/backfill_history.py 40
```

Populates `daily_candles` and pre-computes yesterday's indicator values into
`indicator_cache`. The orchestrator also re-runs this automatically at 09:00 IST.

---

## 5. Start the WhatsApp service (first-time QR scan)

```bash
cd whatsapp-service
node index.js
```

On first run a **QR code** prints in the terminal. On your phone:
**WhatsApp → Settings → Linked Devices → Link a Device** and scan it. The
session is saved in `auth_store/`, so subsequent restarts reconnect silently.

Verify it's up:

```bash
curl http://localhost:3001/health
# {"status":"ok","connected":true,"queued":0}

curl -X POST http://localhost:3001/send-message \
  -H 'Content-Type: application/json' \
  -d '{"phone":"919876543210","message":"test ✅"}'
```

---

## 6. Run the screener

Keep the WhatsApp service running, then in another terminal:

```bash
cd python-engine
source .venv/bin/activate
python main.py
```

The orchestrator schedules everything on **Asia/Kolkata** time:

| Time (IST) | Action |
|-----------|--------|
| 09:00 | Refresh historical candles + pre-compute yesterday's indicators |
| 09:14 | Connect WebSocket, subscribe to all symbols (batches of 1000) |
| 09:15 | Reset forming candles (market open) |
| 09:20 | Start screener loop (every 2 minutes) |
| 15:30 | Stop screener loop (market close) |
| 15:35 | Persist today's final candle as a closed candle |
| 15:40 | Disconnect WebSocket, send daily summary |

If you start it **during** market hours it catches up automatically (prep →
connect → scan loop). Holidays and weekends are skipped (NSE holiday list in
`main.py` → `NSE_HOLIDAYS_2026`; update it yearly from the NSE circular).

### Useful run modes

```bash
python main.py --prep    # run just the 09:00 prep step now
python main.py --once    # auth, connect, wait 20s for ticks, run ONE scan, exit
```

---

## 7. Verify everything is working

- `curl http://localhost:3001/health` → `connected: true`.
- `logs/screener.log` shows `Scan complete: scanned=… matches=…` every 2 min.
- During market hours, `live candles: N` in the logs should grow as ticks arrive.
- A matching stock produces a formatted WhatsApp message and an `alert_log` row.
- Inspect the DB: `sqlite3 database/screener.db "SELECT * FROM alert_log;"`

## Web dashboard (frontend)

A built-in web dashboard visualises everything the screener does: a candlestick
chart **per stock with the indicator lines drawn on it** (9 EMA, Bollinger
Bands, VWAP), a filter to show only the stocks that are **ready to buy**, and a
**Notifications** panel listing the WhatsApp alerts that have been sent.

It runs on Python's standard library (`http.server`) — **no Flask or extra
installs** — so it works with the same venv the engine uses.

```bash
# (optional) seed synthetic candles so you can see it immediately, with no
# broker feed / outside market hours. DEMO DATA ONLY — not real prices.
python scripts/seed_demo_data.py

# start the dashboard
cd python-engine
python web_server.py            # http://localhost:8000  (or: python web_server.py 8080)
```

Open <http://localhost:8000> in a browser.

**What you can do**

- **See every chart** with candlesticks + indicator overlays.
- **Filter** with the top buttons: *All* · *✅ Ready to Buy* (both crossover
  conditions met) · *9 EMA × BB Mid* · *VWAP × BB Upper*. Charts that pass the
  selected filter are the ones shown.
- **Toggle indicator lines** (9 EMA / Bollinger / VWAP) on the charts.
- **Search** by symbol or company name.
- **Notifications drawer** (🔔) lists alerts that fired, with a *SENT/PENDING*
  status, and a **Send test WhatsApp alert** button (requires the
  `whatsapp-service` to be running).

The "ready to buy" signal shown is computed from the daily candles by
`indicator_engine.evaluate_daily_signal` — the same crossover logic the live
screener uses, so the dashboard stays consistent with the alerts that get sent.
It reads from the same `database/screener.db`, so once the real engine
(`main.py`) is running and the DB has live candles, the dashboard reflects them
(it auto-refreshes every 60s).

### Deploy the dashboard to Vercel (static hosting)

The live engine (broker WebSocket, screener loop, WhatsApp service) needs a
long-running process and **cannot** run on Vercel's serverless platform. But the
**dashboard frontend can** — it falls back to pre-exported JSON snapshots when no
Python API is present.

1. Export a snapshot of the current DB to `web/data/`:
   ```bash
   python scripts/seed_demo_data.py      # or backfill_history.py for real data
   python scripts/export_static.py       # writes web/data/{dashboard,alerts,stats}.json
   git add web/data && git commit -m "Update dashboard snapshot" && git push
   ```
2. Import the repo at <https://vercel.com/new>. `vercel.json` already points the
   output directory at `web/`, so no build step or framework preset is needed —
   just deploy.
3. The deployed site reads `web/data/*.json` (the snapshot). Re-run step 1 and
   push to refresh the data shown online. Running `web_server.py` locally still
   uses the live `/api/*` automatically.

> Note: the "Send test WhatsApp alert" button needs the local backend +
> `whatsapp-service`; on the static Vercel deployment it will report the backend
> is unreachable (expected).

#### Why WhatsApp / QR can't run on Vercel

The WhatsApp service is a **stateful, always-on Node process** (Baileys keeps a
live socket + session). Vercel is **serverless** — it can't host a long-running
connection, so on the hosted site the WhatsApp panel shows *"No backend
connected"* with instructions. The QR + sending only work where the backend
runs:

- **Locally** — run `whatsapp-service` + `web_server.py`, open
  `http://localhost:8000`; the QR appears in the 🔔 panel when not linked.
- **Hosted dashboard + your backend** — run the backend somewhere reachable
  (your PC via a tunnel like cloudflared/ngrok, or a VPS/Render) and point the
  hosted page at it by adding `?api=https://your-backend` to the URL (saved to
  `localStorage`). The dashboard will then fetch status/QR/history and send
  through that backend.

#### One command to make the hosted site work (Windows)

```powershell
winget install Cloudflare.cloudflared        # once
powershell -ExecutionPolicy Bypass -File scripts\serve_public.ps1
```

`serve_public.ps1` starts the WhatsApp service + dashboard API + a Cloudflare
quick tunnel, then prints a ready-to-open link:

```
https://screeners-hgg6.vercel.app/?api=https://<random>.trycloudflare.com
```

Open that link — the hosted dashboard now talks to your PC, so WhatsApp shows
**Connected**, charts load, and **Send Ready-to-Buy** works. The tunnel URL
changes each run (quick tunnels are ephemeral); keep the window open while you
use it. The Python server already sends permissive CORS headers so the
cross-origin calls from `*.vercel.app` succeed.

### Dashboard API

| Method | Endpoint | Purpose |
|--------|----------|---------|
| GET | `/api/dashboard?filter=all\|ready\|ema\|vwap&limit=` | All symbols' series + signal |

## Troubleshooting

| Problem | Fix |
|--------|-----|
| `Angel One auth attempt … failed` | Check API key / client ID / MPIN / TOTP secret. The TOTP secret must be the **base32 seed**, not a 6-digit code. Server clock must be accurate (TOTP is time-based). |
| WhatsApp messages not arriving | Is `node index.js` running and `connected: true`? Check `delay_between_messages_sec` and that the phone has no `+`. |
| `whatsapp_not_connected` | Re-scan the QR. If logged out, delete `whatsapp-service/auth_store/` and restart. |
| No ticks / empty candles | Markets must be open (09:15–15:30 IST, weekday, non-holiday). Verify the WebSocket subscribed in the logs. |
| Historical data missing for a symbol | The fetcher falls back to `yfinance` (`SYMBOL.NS`). Some illiquid/odd symbols may have no data and are simply skipped. |
| `smartapi-python` import errors | Ensure `pip install -r requirements.txt` completed; the package imports as `from SmartApi import SmartConnect`. |
| Hitting WhatsApp rate limits / ban risk | Keep the 2s gap (`delay_between_messages_sec`) and the 400/day cap. Don't lower them. |

## Customising the filter

The whole signal lives in `python-engine/screener.py → check_signal()`. To
change it:

- **Different bands/periods:** edit `config.json → screener` (`bollinger_period`,
  `bollinger_std`, `ema_period`).
- **Different crossover logic:** modify the `ema_crossed_bb_mid` /
  `vwap_crossed_bb_upper` expressions, or add new conditions. Current indicator
  values come from `compute_indicators()` (latest row), previous values from the
  cached `yesterday_values`.
- **New indicators:** add a function in `indicator_engine.py`, compute it in
  `compute_indicators()`, and reference it in `check_signal()`. If you need a
  cached previous value, also store it in `compute_yesterday_indicators()` and
  the `indicator_cache` table (`db.py`).
- **Scan frequency:** `config.json → screener.scan_interval_minutes`.
- **Alert format:** `alert_manager.py → format_alert()`.

---

## Notes & caveats

- VWAP is a true intraday measure; for the *previous* day we use the typical
  price `(H+L+C)/3` of yesterday's closed candle as `vwap_proxy` for the
  crossover comparison (intraday tick history isn't retained overnight).
- Subscribing to thousands of symbols at full quote depth is heavy; Angel One
  enforces per-connection limits. If you hit them, reduce the symbol list or
  raise `websocket_subscribe_batch` pacing.
- This is a screening/alert tool, **not** trading advice or an order-execution
  system. Use at your own risk.
